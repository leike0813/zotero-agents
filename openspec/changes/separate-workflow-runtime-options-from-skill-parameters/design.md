# Design: Workflow Runtime-Only Parameters

## Decision

Workflow parameter declarations may set `runtimeOnly: true`. The setting remains part of the workflow settings UI and run submission state, but it is not a provider-facing skill parameter.

This keeps a single user configuration surface while establishing a clear execution boundary:

- normal parameters are skill/provider input;
- runtime-only parameters are Zotero Skills local execution metadata.

## Parameter Split

At request-build time, the runtime resolves workflow settings once, then splits the resolved values by manifest declaration:

- `providerParameters`: parameters without `runtimeOnly: true`;
- `workflowRuntimeParams`: parameters with `runtimeOnly: true`.

Provider requests use only `providerParameters` for `request.parameter`. `workflowRuntimeParams` stays in the local execution context and is passed to apply hooks through `resultContext`.

## Apply Hook Access

Apply hooks read runtime-only values from `args.resultContext.workflowRuntimeParams`. Hooks may keep backward-compatible fallbacks for old run records, but new runs must not depend on provider-visible request fields for runtime-only behavior.

## Literature Digest

`auto_reference_matching` is a runtime-only parameter because it controls a local apply-time post-process after digest notes are written. The skill does not need to know about it, and exposing it to the agent can confuse the skill contract.

`language` remains a normal parameter because it affects the digest output generated by the skill.

## Compatibility

Remote SkillRunner requests and ACP skill requests follow the same split. This avoids a backend-specific exception and keeps provider contracts honest: runtime-only parameters are never part of public provider dispatch payloads.
