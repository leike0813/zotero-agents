## Context

The UI optimization summary at `artifact/ui_optimization_summary_20260613.md`
documents visual and harness work completed in the `explore-harness-ui-code`
worktree. The affected surfaces are not one isolated component: the work spans
the shared Assistant panel renderer, Dashboard workflow cards, Workflow
Settings dialog styling, Synthesis Topic Detail styling, shared theme tokens,
and readonly harness screenshot tooling.

The implementation should remain UI-only. It must not change ACP protocol
payloads, workflow execution semantics, Synthesis storage contracts, Zotero
host commands, or backend state.

## Goals / Non-Goals

**Goals:**

- Promote the artifact's UI improvements into OpenSpec requirements so future
  changes preserve compact alignment, readable spacing, and shared token usage.
- Define how readonly harness visual verification should wait for real mounted
  DOM state before taking screenshots.
- Keep runtime artifacts generated by visual verification out of source unless
  they are explicitly promoted as documented test fixtures.
- Make the follow-up traceable to the artifact and to the files modified by the
  worktree.

**Non-Goals:**

- No new UI framework, rendering architecture, or browser automation dependency.
- No changes to ACP run state, SkillRunner observation, workflow execution, or
  Synthesis persistence contracts.
- No requirement to store generated screenshots in the repository.
- No requirement to start the harness or dev server during normal validation.

## Decisions

### Decision: Treat the summary artifact as the traceability anchor

The change SHALL cite `artifact/ui_optimization_summary_20260613.md` as the
source of truth for the follow-up scope. This avoids expanding the change into
unrelated UI redesign while still preserving the rationale for the optimized
surfaces.

Alternative considered: infer scope only from current Git diff. That would lose
the screenshot/CDP harness context and make it harder to distinguish source
changes from generated runtime artifacts.

### Decision: Add focused requirements to existing capabilities

The follow-up adds requirements to `zotero-skills-visual-theme`,
`assistant-sidebar-ui`, `synthesis-tab-ui`, and `ui-readonly-harness`.
Workflow Settings dialog layout remains an impact area under the shared visual
theme because the current `workflow-settings-dialog-model` capability describes
descriptor behavior, not visual layout.

Alternative considered: create a new capability for UI polish. That would
fragment contracts already owned by the shared theme, Assistant, Synthesis, and
harness specs.

### Decision: Use DOM readiness, not fixed sleeps, for visual capture

Readonly harness screenshot tooling SHALL wait for expected page roots and
surface content before capturing. Fixed sleeps are not robust when the local
harness is blocked by SQLite reads, bundle rebuilds, or slow browser startup.

Alternative considered: keep longer fixed timeouts. That makes failures slower
and still does not prove the target surface mounted.

### Decision: Keep generated visual artifacts out of source by default

Browser profile directories, local mock data, SQLite databases, and screenshot
outputs are verification artifacts. They remain excluded from source unless a
future task explicitly promotes a small, stable fixture with a documented
purpose.

Alternative considered: commit all generated screenshots. That would add churn
without creating stable behavioral coverage.

## Risks / Trade-offs

- Visual requirements can become too brittle if they lock exact pixel values.
  Mitigation: specify observable layout behavior and token families rather than
  full pixel snapshots.
- Harness readiness checks may miss a new surface if selectors drift.
  Mitigation: keep readiness tied to stable page roots and domain-level content
  instead of decorative CSS selectors.
- Some optimization work was already applied before this follow-up.
  Mitigation: tasks include auditing the current implementation against the
  new requirements and keeping validation focused.

## Migration Plan

1. Keep the already-imported UI source changes in place.
2. Add or adjust focused tests only where an existing stable user-visible
   behavior is not already covered.
3. Run strict OpenSpec validation for this change.
4. Run targeted UI/core tests and `npm run build` when implementation changes
   are applied or rechecked.

- Should Workflow Settings dialog layout eventually get its own visual
