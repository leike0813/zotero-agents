# Отладка и тестирование

После написания пользовательского Workflow вы можете использовать следующие методы для его проверки и отладки.

## Включение режима отладки

Включите режим отладки в настройках, чтобы разблокировать дополнительные инструменты отладки и отображение информации:

Zotero → Настройки → Zotero Agents → Включить режим отладки

Когда режим отладки включён:

- Workflow, связанные с отладкой, отображаются в Панели мониторинга
- Логи выполнения становятся более подробными
- Некоторые диагностические инструменты становятся доступными

## Использование набора инструментов Debug Probe

Плагин включает встроенный набор инструментов отладки `workflow-debug-probe`, содержащий несколько диагностических Workflow:

| Workflow | Назначение |
|----------|---------|
| **Workflow Debug Probe** | Исследовать состояние Workflow перед выполнением, открыть диагностическую панель |
| **Debug Sequence Linear Probe** | Проверить последовательное выполнение и передачу handoff по умолчанию |
| **Debug Sequence Workspace Reuse Probe** | Проверить повторное использование рабочего пространства между шагами |
| **Debug Sequence Context Isolation Probe** | Проверить явную фильтрацию handoff и изолированные рабочие пространства |

Эти Workflow видны в списке Workflow Панели мониторинга (в режиме отладки) и могут быть запущены напрямую для проверки механизмов выполнения последовательности.

## Просмотр логов

### Логи выполнения

Workflow генерируют логи выполнения во время выполнения, просматриваемые в Панели мониторинга:

1. Откройте Панель мониторинга
2. Найдите выполняющуюся или завершённую задачу
3. Нажмите "Просмотреть логи", чтобы развернуть панель логов

### Написание логов в Hook

```js
export function applyResult({ parent, bundleReader, runtime }) {
  // Записать в лог выполнения
  runtime.hostApi.logging.appendRuntimeLog({
    level: "info",
    message: `Processing parent: ${parent}`,
    workflowId: runtime.workflowId,
  });

  // Для сложной диагностической информации вы можете использовать console
  console.log("Debug:", { parent, workflowId: runtime.workflowId });
}
```

## Устранение распространённых проблем

### Workflow не отображается в Панели мониторинга

1. Проверьте, размещён ли `workflow.json` в правильном каталоге
2. Подтвердите, что `workflow.json` имеет правильный формат (синтаксис JSON)
3. Проверьте, что `id` уникален и не конфликтует с официальными Workflow
4. Подтвердите, что путь к скрипту `applyResult` корректен
5. Проверьте лог ошибок плагина (Zotero → Справка → Устранение неполадок → Просмотреть файл лога)

### filterInputs возвращает null

Если `filterInputs` возвращает `null`, это означает, что не было найдено подходящего выбора, и Workflow не будет выполняться. Проверьте, правильна ли логика фильтрации.

### Конфликт между buildRequest и декларативным запросом

Hook `buildRequest` и поле `request` в `workflow.json` **взаимоисключающие**. Если оба существуют, `buildRequest` имеет приоритет. Если поведение запроса не такое, как ожидалось, проверьте, не были ли оба определены непреднамеренно одновременно.

### Сбой выполнения скрипта Hook

- Подтвердите, что скрипт Hook имеет формат `.mjs` (ES Module)
- Подтвердите, что экспортированы правильные имена функций: `filterInputs`, `buildRequest`, `applyResult`
- Подтвердите, что сигнатура функции корректно принимает параметры, такие как `{ parent, bundleReader, runtime }`
- Проверьте, правильны ли пути относительного импорта

### Результат не записан в Zotero

Если `applyResult` использует `hostApi.mutations.execute()`, но это не вступает в силу, возможные причины:

- Операции записи требуют подтверждения от пользователя, но всплывающее окно подтверждения было проигнорировано или время истекло
- Попытка выполнить операцию записи, когда `execution.zoteroHostAccess.required` не было установлено в `true`
- `allowWriteApprovalBypass` нужно использовать вместе с конфигурацией разрешений плагина

## Предложения по разработке

### Начните с простого

1. Сначала используйте провайдер `pass-through` с минимальным `applyResult`, чтобы проверить, что Workflow загружается успешно
2. Постепенно добавляйте `filterInputs` и `buildRequest`
3. Наконец, подключитесь к фактическому бэкенду

### Используйте notifications.toast для быстрой обратной связи

```js
hostApi.notifications.toast({
  text: `filterInputs received ${selectionContext.items.parents.length} parent items`,
  type: "default",
});
```

Это быстрый метод отладки, который позволяет вам видеть результаты выполнения без проверки логов.

### Справьтесь с официальными Workflow

Официальные Workflow являются лучшим справочным материалом для обучения. После установки официального пакета вы можете просмотреть исходный код в каталоге `<Данные Zotero>/zotero-agents/content/official/workflows/`:

- `literature-workbench-package/literature-analysis/` — Полный пример skillrunner.job.v1
- `content/official/workflows/literature-workbench-package/export-notes/` — Простой пример pass-through
- `content/official/workflows/mineru/` — Пример с buildRequest + обработкой файлов
- `content/official/workflows/literature-workbench-package/literature-search-ingest/` — Пример интерактивного режима
